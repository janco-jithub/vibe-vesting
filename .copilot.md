# Project: Quantitative Trading System

## Overview

This is a professional-grade quantitative trading system focused on maximizing risk-adjusted returns using proven, academically-validated strategies. All code must prioritize correctness over cleverness, and every trading decision must be backed by robust risk management.

## Tech Stack

| Component | Technology | Purpose |
|-----------|------------|---------|
| Language | Python 3.11+ | Core development |
| Backtesting | VectorBT / Backtrader | Strategy validation |
| Data Storage | TimescaleDB | Time-series data |
| Market Data | Polygon.io / Alpaca | Real-time & historical |
| ML Framework | scikit-learn, XGBoost, LightGBM | Predictive models |
| Feature Engineering | pandas-ta, TA-Lib | Technical indicators |
| Execution | Alpaca API / Interactive Brokers | Order management |
| Monitoring | Grafana + Prometheus | Performance dashboards |
| Testing | pytest | Unit and integration tests |

## Coding Standards

### General
- Type hints required on all functions
- Docstrings required on all public functions and classes
- All numeric operations use `Decimal` or `numpy` with explicit precision
- No magic numbers—all constants must be named and documented

### Testing Requirements
- Unit test coverage > 90%
- All strategies must have backtest validation tests
- Integration tests for broker connectivity
- Mock external APIs in unit tests

### Backtesting Requirements
- Transaction costs: minimum 10 bps (0.1%)
- Slippage model: volume-based or fixed percentage
- No look-ahead bias allowed
- Walk-forward validation required

### Risk Limits (Enforced in Code)
- Maximum single position: 5% of portfolio
- Maximum sector exposure: 25% of portfolio
- Daily loss limit: -2%
- Maximum drawdown halt: -15%
- All positions must have stop-loss

## Important Patterns

### Strategy Implementation
All strategies must inherit from `BaseStrategy` and implement:
- `generate_signals(data) -> pd.DataFrame`
- `calculate_position_size(signal, portfolio) -> float`
- `get_backtest_params() -> dict`

### Risk Checks
Every order must pass through the risk manager:
```python
if not risk_manager.validate_order(order):
    raise RiskLimitExceeded(risk_manager.get_violation_reason())
```

### Data Validation
All market data must be validated before use:
- Check for missing values
- Verify price continuity (no >20% gaps without corporate actions)
- Confirm timezone consistency (UTC internally)

### Logging
Use structured logging for all trading activity:
```python
logger.info("order_placed", order_id=order.id, symbol=order.symbol, quantity=order.quantity)
```

## What NOT to Do

### Trading Safety
- ❌ Never use market orders in live trading
- ❌ Never trade without stop-loss mechanisms
- ❌ Never deploy without paper trading validation (min 3 months)
- ❌ Never exceed position size limits
- ❌ Never ignore circuit breakers

### Code Quality
- ❌ Never use random train/test splits for time-series (use walk-forward)
- ❌ Never hardcode API keys (use environment variables)
- ❌ Never commit sensitive data to git
- ❌ Never use `float` for money calculations (use `Decimal`)

### Backtesting Integrity
- ❌ Never use future data in feature calculation (look-ahead bias)
- ❌ Never ignore transaction costs and slippage
- ❌ Never use survivorship-biased data
- ❌ Never optimize on test data
